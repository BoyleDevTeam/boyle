## 10. 测试与迭代

### 10.1 评估驱动开发

**在编写详细文档之前先创建评估场景。** 确保 Skill 解决的是真实问题而非想象中的问题。

1. **识别差距**：在没有 Skill 的情况下让 Agent 执行代表性任务，记录具体的失败
2. **创建评估**：设计 3 个测试场景覆盖这些差距
3. **建立基线**：测量没有 Skill 时 Agent 的表现
4. **编写最小指令**：只写足以解决差距的内容
5. **迭代**：执行评估，与基线对比，持续改进

### 10.2 与 Agent 协作开发

最有效的 Skill 开发流程涉及 Agent 本身：

1. **完成一个任务**（不用 Skill）：在正常对话中解决问题，注意你反复提供的上下文
2. **识别可复用模式**：哪些上下文对未来类似任务有用？
3. **让 Agent 创建 Skill**：基于对话模式生成 Skill 草案
4. **审查简洁性**：删除不必要的解释
5. **测试**：在新的对话中用 Skill 执行相似任务
6. **迭代改进**：根据观察的 Agent 行为调整

### 10.3 观察 Agent 如何使用 Skill

关注 Agent 实际使用 Skill 的方式：

- **意外的浏览路径**：Agent 是否按你预期的顺序读取文件？
- **遗漏的引用**：Agent 是否跟随了重要文件的链接？
- **过度依赖某部分**：某个文件被反复阅读？考虑将其内容放入 `SKILL.md`
- **忽略的内容**：某个辅助文件从未被访问？它可能是多余的

### 10.4 纪律性 Skill 的压力测试

对于强制 Agent 遵守流程的 Skill（如 TDD、调试流程），需要进行压力测试：

1. **基线测试**：不加载 Skill，给 Agent 施加时间压力，记录它使用的"合理化借口"
2. **加载 Skill 测试**：同样的压力场景，验证 Agent 是否遵守
3. **封堵漏洞**：发现新的绕过方式时，添加明确的反制措施
4. **循环**：直到 Agent 在最大压力下仍遵守规则
